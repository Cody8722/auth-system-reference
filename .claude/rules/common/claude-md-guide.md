# CLAUDE.md 撰寫規範

好的 CLAUDE.md 不只是「文件」——它是給 Claude 的**操作手冊**，讓每次新對話都能立刻進入狀況，同時防止 Claude 做出危險或不符預期的操作。

每個專案都應該在根目錄維護一份 CLAUDE.md。以下是各區塊的內容規範。

---

## 操作規則（必填）

這是「給 Claude 的行為約束」，所有專案都必須包含以下規則，不可省略：

- 禁止單獨使用 `cd`，優先使用相對路徑
- 禁止使用 `cd ... && <指令>` 的組合，直接用 `<指令> path/to/target`
- 避免複雜 Bash 組合：如需複雜操作，請分開多次執行
- 每個 bash 呼叫是獨立 session，不依賴上一條指令的環境變數
- 禁止使用 `sleep` 等待非同步任務；等待 GitHub Action 應改用 `gh run watch`
- 使用者明確要求刪除或移動檔案時，直接執行即可，無需再反問確認一次
- 修改檔案前必須先完整讀過，不靠假設
- 不刪除未觸及的程式碼、注解、TODO
- 不主動重構超出需求範圍的程式碼
- 遇到不確定的需求先問，不自行假設後執行
- 失敗時回報具體錯誤訊息與指令，不要空泛說「發生錯誤」
- 不可覆寫或修改現有的 `.env` 檔案；但可以新增 `.env.example` 或 `.env.template`
- 不可為了讓測試通過而修改測試邏輯，應修正程式碼本身

根據專案環境額外加入：

- 有 git：commit 格式規範、分支策略、何時 push
- 有 CI/CD：push 前（而非每次修改後）必須確認測試通過
- 有資料庫：schema 變更規則、migration 流程
- 有前端：框架慣例；若有 service worker，說明版本號是否需手動維護

---

## Commit 規範（有 git 時必填）

格式：`<類型>: <簡短描述>`

| 類型 | 用途 |
|------|------|
| `feat` | 新功能 |
| `fix` | Bug 修復 |
| `refactor` | 重構（非新功能、非 bug）|
| `docs` | 文件變更 |
| `test` | 測試相關 |
| `perf` | 效能優化 |
| `chore` | 建置工具或雜項變更 |
| `ci` | CI/CD 設定變更 |
| `style` | 格式調整（不影響邏輯）|

每約 5 個相關變更，或一個功能階段完成後才 commit。commit 完成後必須立即推送，不可只 commit 不 push。

---

## 分支策略（有 git 時必填）

```
feature/* → develop（測試版） → release（穩定版，透過 PR）
```

- `feature/<name>`：所有新功能與 bug 修復在此開發
- `develop`：整合分支，推送後 CI 自動觸發測試；直接 push 僅限小修（文件、設定）
- `release`：禁止直接 push，必須從 `develop` 開 PR 合併

---

## 向下兼容原則（有 API 或資料庫時必填）

- **API**：現有端點的參數、回應格式不可破壞；新增參數必須有預設值
- **資料庫**：不可刪除或重命名現有欄位；新增欄位必須有預設值或允許 null
- **前端**：不可移除現有的 HTML id/class（若其他模組有依賴）；JS 公開介面不可改名
- **測試**：現有測試不可因重構而失敗；新功能必須補上對應測試
- 如需破壞性變更：必須先與使用者確認，並在 CHANGELOG.md 標注為 Breaking Change

---

## 架構總覽

不只是目錄結構——要說明**系統各部分如何協作**。範例：

```
frontend/（純靜態 HTML + Vanilla JS）
    ↕ HTTP Fetch API (JWT Bearer Token)
backend/（Flask Python API）
    ↕ PyMongo
MongoDB（accounting_db）
```

讓 Claude 在修改某個部分時，知道可能影響到哪些地方。

---

## 常用指令

從 `package.json` scripts、`Makefile`、README 提取真實指令，每個指令附上用途說明。

分類建議：開發伺服器、建置、測試、Lint / 格式化、資料庫操作、部署。

**推送前必須執行**這個子區塊尤其重要——明確列出 `git push` 前的最終檢查清單，讓 Claude 知道「功能完成準備推送時」才需要跑完整測試，而不是每改一個檔案就跑一次：

```bash
# push 前執行
npm run lint && npm test && npm run build
```

---

## 特定操作的必做事項

如果有「修改某類檔案後必須額外執行的動作」，獨立成區塊明確標注。例如：

- 修改前端後，必須更新 `service-worker.js` 的版本號
- 修改 schema 後，必須跑 migration
- 更新 API 後，必須同步更新 `docs/API.md`

---

## 程式碼風格

只寫這個專案的特殊規定，不寫通用建議。例如：

- 使用的格式化工具及設定（Black 88 字元、Prettier 等）
- 命名慣例（camelCase vs snake_case 的使用場合）
- 禁止事項（禁用 `var`、禁止 `console.log` 等）
- 特殊的 import 規則或模組架構慣例

---

## 已知地雷 / 歷史包袱（最有價值的區塊）

記錄曾經踩過的坑、違反直覺的設計、隱藏的依賴關係。這是最難自動偵測的資訊，也是讓 Claude 最快上手的關鍵。範例：

- 某個欄位的資料型別和其他集合不一致
- 環境變數有特殊要求（必填、測試環境的特殊設定）
- 某個功能依賴特定的執行順序或前置條件
- 前端某個全域函式不可改名（有外部依賴）

每次踩到新的坑，記得更新這裡。

---

## 環境變數（有 .env 時）

整理成表格，標注哪些是必填、用途是什麼、測試環境的最小設定。

## 相關文件

列出其他重要文件的連結和一句話說明，讓 Claude 知道需要更多細節時去哪裡找。
